## Architecture Patterns

### Module exports
Shared application modules (`imports/`) use named exports rather than default exports so imports stay explicit and refactors are easier to trace. Ambient declarations for untyped NPM packages may still describe a default export for that package's runtime shape.

### Publications and Observers
- Server publications commonly wrap Mongo observers using a `registerObserver` utility (`imports/server/methods.ts`) that returns `{ added, changed, removed }` handlers with a shared transformer.
- Publications use `Tracker.autorun` to reactively derive IDs from one collection (e.g., `MemberThemes`) and observe another (`Members`), and tie observer/computation cleanup to the publication via `this.onStop`.
- Tracker server reactivity still relies on synchronous reads; use `find()` / `findOne()` / `fetch()` inside the autorun and avoid awaiting `fetchAsync` or `findOneAsync` in the reactive block, otherwise downstream subscribers stop receiving updates.
- To keep Meteor 3 async fetches reactive, publications now pair lightweight `observeChanges` watchers with a single live query handle; watchers trigger a debounced async refresh that refetches the dependent data and restarts the shared observer without spawning duplicate handles.
- `MemberThemes` updates for a theme are coordinated in `imports/server/publications/memberThemesRefreshCoordinator.ts`: one `observeChanges` + debounced fetch per `themeId`, shared by the `theme`, `organizations`, and `members` publications so concurrent subscribers do not multiply Mongo watchers or redundant full-table fetches.
- Publications that derive membership through join tables must diff published IDs before recreating their observers (`imports/server/publications/members.ts`). The diff explicitly emits `removed` messages for IDs dropped from the join table and reuses the shared transformer so downstream UI collections immediately drop stale rows when the join changes.

### Pledge Matching Pipeline (Leverage Pool Walk)
- All matched-pledge math lives in `imports/lib/pledgeMatching.ts`. **Never** open-code `pledge.amount * matchRatio` anywhere else - it ignores the leverage pool, mixed `pledgeType` ratios, and partial-match truncation.
- There is one shared leverage pool per theme. All eligible orgs' pledges are merged into a single chronological queue (`createdAt`, then `_id` for tie-break) and consume that one pool in order. A pledge that lands late may receive a partial match â€” or none at all â€” if earlier pledges to *any* org have already drained the pool.
- `calculatePledgeMatches` returns `{ matchedAmounts: Map<pledgeId, number>, remainingLeverage }` and **persists nothing**. Derived values (`leverageBonus`, org `pledgeTotal`, `theme.leverageRemaining`) are recomputed on every publication tick. This keeps state safe when an earlier pledge is removed/edited - the whole walk re-runs and downstream pledges are promoted/demoted automatically.
- `MatchPledge.pledgeType` is the discriminator (`"standard"` | `"inPerson"`). Standard uses `Theme.matchRatio`; in-person uses `Theme.inPersonMatchRatio`. Both ratios are **total multipliers**: ratio 2 = 1:1 leverage match (pledge $1, leverage adds $1), ratio 3 = 2:1 leverage match.
- `OrgTransformer` accepts `params.matchedAmounts`. When omitted (standalone use, isolated unit tests), it falls back to "fully matched" so legacy callers still get `amount * matchRatio` semantics. The actual per-org contribution math lives in `pledgeTotalForOrg` inside `pledgeMatching.ts` â€” the transformer just delegates so the runner-up eligibility rule and matched-amount fallback have one definition.
- Runner-up eligibility (a finalist always counts; a runner-up only when `theme.leverageRunnersUpPledges` is true) is a single helper, `isOrgEligibleForLeverage`, used by both `calculatePledgeMatches` (when building the chronological queue) and `pledgeTotalForOrg` (when summing an org's contribution). Don't duplicate it.
- `ThemeTransformer` exposes two related-but-distinct totals: `theme.pledgedTotal` is the **raw** sum of pledge amounts (used by the Pledges admin pane's footer to match the per-org rows above it); `theme.pledgeMatchTotal` is the total **leverage bonus consumed** by pledges (used in `AllocationPane/Breakdown` so the displayed `Starting Leverage âˆ’ X = Remaining` equation balances under non-2 ratios and partial matches). They only coincide when every pledge is standard with `matchRatio === 2` and the pool was never exhausted.
- Publications must thread `matchedAmounts` through every transformer call. The `themes` publication computes it once via `computePledgeMatchingForPublication(...)` and passes it into both `OrgTransformer` (per-org `pledgeTotal`) and `ThemeTransformer` (`leverageRemaining`). The `organizations` publication does the same and **wraps the org `observe()` callbacks with a debounced full re-publish** because a single org change shifts the chronological walk for every other org.
- All in-person pledge configuration (`inPersonMatchRatio`, `inPersonPledgeActive`) lives on the **Theme** schema, not `PresentationSettings`. The match ratio and the feature flag that enables the variant are one logical group and travel together.
- The in-person pledge route is `/pledges/inperson/:themeId` (`InPersonPledgeRoute`). It uses the theme's Mongo `_id` rather than `slug` so the URL is unguessable enough to discourage casual sharing for the ~30-minute in-person pledge window. This is **obscurity, not security** â€” the `_id` is not a rotatable secret. The QR code is **printable-only**: it lives in `PresentationSettingsForm` as a convenience block, gated on `theme.inPersonPledgeActive` (i.e. shown only when the admin has enabled the in-person variant for this theme). It must never render on the presentation view itself.
- `useFindMemberByCode` in `imports/api/hooks` is the shared initials+number lookup used by both the kiosk login flow and the in-person pledge login. Use it instead of duplicating the `code = initials.toUpperCase() + number` calculation.

### Feature Flags vs Runtime Activation
Two distinct concerns; both can stack on top of each other without conflict:
- **Feature flags** live in `ThemeSettingsForm` (Theme schema) and `PresentationSettingsForm` (PresentationSettings configuration fields like `useKioskChitVoting`). They answer "Is this feature available on this theme/presentation?" and are set ahead of time during admin configuration. Examples: `Theme.consolationActive`, `Theme.minLeverageAmountActive`, `Theme.inPersonPledgeActive`, `PresentationSettings.useKioskChitVoting`.
- **Runtime activations** live in `PresentationPane` (the live show control panel) and write to `PresentationSettings` fields that the running presentation observes. They answer "Is this feature being driven right now?" Examples: `topupsActive` (Pledges Active toggle), `chitVotingActive`, `fundsVotingActive`, `currentPage`.
- The same `PresentationSettings` collection holds both feature-flag fields and runtime fields - they're distinguished by which admin UI surfaces them, not by which schema they live in.
- For in-person pledges specifically: `theme.inPersonPledgeActive` is the feature flag (admin enables the variant for this theme), `settings.topupsActive` is the runtime activation (pledges are being accepted right now). The `/pledges/inperson/:themeId` URL is functional only when BOTH are true, and surfaces distinct messaging for each false case.
- Server-side gate for in-person pledges in `OrganizationMethods.pledge` enforces the **feature flag** only (`theme.inPersonPledgeActive`), not the runtime activation - mirroring the standard pledge flow, which also relies on client-side gating for `topupsActive`. This must stay server-only (`Meteor.isServer` guard) to skip method simulation on the client.

### Terminology: pledge / crowdFavorite / leverage
- The legacy codebase used "topup" and "topoff" interchangeably for several distinct concepts. The current vocabulary is:
  - **pledge** - a member's matched donation toward an org during the pledge round.
  - **crowdFavorite** - the one org admins choose to fully fund; the dollar amount needed to bring it to its `ask` is persisted in `Organization.topOff` (legacy field name) and applied via `OrganizationMethods.crowdFavorite`.
  - **leverage** - the matching pool. `Theme.leverageTotal` seeds it; pledges and the crowd-favorite full-funding consume it.
- **Persisted field names kept for backwards compatibility**: `Organization.topOff`, `PresentationSettings.topupsActive`, `PresentationSettings.topupEmailConfirm`. Each has a `// NOTE` comment on its schema definition. Renaming requires a one-time Mongo migration.
- `MinimumLeverageButton` and `CrowdFavoriteButton` were split out of the old overloaded `TopOffButton`: the array-form was for the minimum-leverage feature, the single-org form was for the crowd favorite.

  1. New pledge from database â†’ inserted via Method on first page load

### Method Parameter Typing Pattern
- Leverage existing schema types using TypeScript utility types (`Partial<T>`, `Pick<T, K>`)
- Extend schema types for method-specific requirements rather than duplicating field definitions
- Use `Partial<Member>` for optional member fields, `Pick<Member, "field1" | "field2">` for required fields
- Create minimal interfaces that extend schema types only when additional fields are needed
- Define return types explicitly for all methods to ensure type safety
- Use proper error handling with typed callbacks and Promise rejection patterns

### NEVER USE `any` - CRITICAL RULE
- **ABSOLUTELY FORBIDDEN**: Never use `any` as a type definition
- **NEVER EVER**: No exceptions, no shortcuts, no "just this once"
- **ALWAYS**: Use proper type assertions, union types, or generic constraints instead
- **IF STUCK**: Use `unknown` and narrow the type, or create proper interfaces
- **TYPE ASSERTIONS**: Use `as unknown as SpecificType` when necessary, never `as any`
- **GENERIC CONSTRAINTS**: Use proper generic bounds instead of `any`
- **INTERSECTION TYPES**: Use `TypeA & TypeB` for combining types
- **MAPPED TYPES**: Use utility types like `Partial<T>`, `Pick<T, K>`, `Omit<T, K>`
- **UNION TYPES**: Use `string | number` instead of `any`
- **RECORD TYPES**: Use `Record<string, unknown>` instead of `any`

### NO TYPE CASTING - CRITICAL RULE
- **ABSOLUTELY FORBIDDEN**: Type casting is NOT an acceptable way to handle TypeScript errors
- **NEVER CAST**: Do not use `as` to force types - this defeats the purpose of TypeScript
- **FIX THE ROOT CAUSE**: If you need to cast, the types are wrong - fix the actual type definitions
- **PROPER SOLUTIONS**: 
  - Fix the source type definitions to be correct
  - Use proper generic constraints
  - Create proper interfaces that match the actual data structure
  - Use type guards with `unknown` and narrowing
  - Use intersection types to combine existing types
- **WHEN STUCK**: If you think you need to cast, step back and fix the underlying type system
- **TYPE ASSERTIONS ONLY**: The only acceptable use of `as` is for type assertions with `unknown` when you've properly narrowed the type
- **NO SHORTCUTS**: Type casting is a shortcut that breaks type safety - never use it

### Provider Data Loading Pattern
- Providers should start with data loading enabled by default to avoid race conditions
- `MembersProvider` starts with `subLimit: false` to immediately load members when `themeId` is available
- Remove `useEffect` calls that trigger data loading in provider hooks - let the provider handle loading automatically
- Server publications handle the actual data filtering (e.g., theme-based member filtering)
- Client-side queries should be simple since the server publication does the heavy lifting

### Data Loading Race Condition Prevention
- **Critical Fix**: Publications now fetch and publish initial data synchronously BEFORE setting up observers and calling `this.ready()`. This ensures `subscription.ready()` means data is actually synced to the client.
- **Publication Pattern**: 
  1. Fetch initial data with `fetchAsync()`
  2. Publish it synchronously using observer callbacks (e.g., `observerCallbacks.added(doc)`)
  3. Then set up the observer for future changes
  4. Only then call `this.ready()`
- This eliminates the race condition where `ready()` fired before observer callbacks completed
- Collections must be initialized synchronously when subscriptions become ready, not only in `onReady` callbacks
- Server publications in `Tracker.autorun` must call `this.ready()` in ALL code paths, including early returns when data doesn't exist
- Publications that return early without data must explicitly call `this.ready()` to prevent endless loading states

### Meteor File Loading Behavior
- Meteor eagerly loads and executes any `.js/.ts` files at the project root (outside of `imports/`, `client/`, and `server/`).
- Utility scripts placed in `scripts/` at the root will be executed on server startup if they have top-level code.
- To avoid accidental execution:
  - Place scripts under `imports/` (and only import when needed), or `private/` if treated as assets; and/or
  - Ensure scripts have no top-level side effects and gate execution behind an environment variable check.

### Component Props Typing

### Nested Array Element `_id` autoValue
- For SimpleSchema sub-schemas embedded in array fields (e.g. `MatchPledgeSchema._id`), the `_id` `autoValue` must **only** act on `$push` to assign a fresh `Random.id()`; it must not call `this.unset()` on any other operator.
- `autoValue` runs against the value object inside `$pull` filters too. A `this.unset()` else-branch strips `_id` from the filter, turning `$pull: { pledges: { _id: x } }` into `$pull: { pledges: {} }`, which removes every element. This is silent in single-element arrays but catastrophic for any array with more than one entry.

### Schema Type Generation
- Type generation script (`scripts/generateTypes.ts`) distinguishes between collection-level and embedded schemas
- Collection-level schemas (used with `.attachSchema()` in `index.ts`) automatically get `_id: string` added to their generated types
- Embedded schemas (like `RoundsSchema`, `OrgChitVoteSummary`) do NOT get `_id` added - they're embedded documents without their own identity
- The script parses `imports/api/db/index.ts` to identify which schemas are attached to Mongo collections
- `OrgChitVoteSummary` is an embedded document type (used in Organization.chitVotes), distinct from `ChitVote` (used in MemberTheme.chitVotes)
- When schemas with the same base name exist in different files, ensure they serve different purposes and have unique type names

### Chit Voting System Architecture
The application supports two modes of chit voting controlled by `PresentationSettings.useKioskChitVoting`r:

**Manual Mode (useKioskChitVoting = false)**:
- Historical mode for physical wooden chits placed in boxes
- Admins weigh/count chits and manually enter data in `ManualInputTable`
- Data stored in `Organization.chitVotes` as `OrgChitVoteSummary { weight?: number, count?: number }`
- `count` takes precedence over `weight` when both present
- `weight` divided by `Theme.chitWeight` to calculate vote count
- One summary record per organization

**Kiosk Mode (useKioskChitVoting = true)**:
- Digital voting through the application interface
- Members distribute chits across organizations using `ChitTicker` component
- Data stored in `MemberTheme.chitVotes[]` as array of `ChitVote { organization, votes, voteSource, createdAt }`
- One `ChitVote` record per organization per member
- `OrgTransformer` aggregates votes from all member records

The two data structures serve different purposes and should not be conflated:
- `OrgChitVoteSummary`: Aggregate summary for manual entry (no _id needed)
- `ChitVote`: Individual vote records for kiosk mode (needs _id for array updates)

### Test Database Reset Helper
- Unit and integration tests reset their database state through `imports/test-support/resetDatabase`, which clears each app collection with `removeAsync({})`. ### Logging Architecture
- Structured logger lives in `imports/lib/logger.ts`. Methods under `imports/api/methods/` ship to the client for optimistic UI, so any utility they depend on must live outside `imports/server/` â€” Meteor enforces a client-cannot-import-server boundary at bundle time.
- The logger is **exclusively for events that happen during a presentation** â€” i.e. scoped to a `themeId`. `LogContext` makes `themeId` required on every `log.info/warn/error/debug` call, and `LogSchema.themeId` is also `required: true`. App-level events (server startup, scheduled jobs, cross-theme admin operations like member deletion or `logs.purge`) go to `console.*` directly. If you have no `themeId` for what you're observing, this isn't the right tool.
- Pre-configured loggers are exported by name from `imports/lib/loggers.ts` (`smsLog`, `emailLog`, `memberMethodLog`, `organizationMethodLog`). Consumers import the one they need; one-off loggers can be built via `createLogger` from `imports/lib/logger.ts`. There is intentionally no generic "method" logger â€” `model` is part of each named logger's identity, so each `*Methods.ts` file gets its own.
- `createLogger` takes a single `LoggerSettings` object (`{ category, model?, meta? }`). Console mirroring is per-call, opt-in via `mirrorToConsole: true` on the `LogContext` passed to `log.info/warn/error/debug` â€” useful for selectively echoing a specific event during debugging. Persist failures still go to `console.error` unconditionally because that's the logger reporting it can't do its job. DB writes are guarded by `Meteor.isServer`; client-side optimistic method runs are silent (and never attempt a privileged insert).
- `Logs` collection denies all client writes via `logsPermissions`. Inserts happen only through the server-side logger, and pruning is gated by the `logs.purge` method (`{ themeId }` â€” wipes all logs for the given theme; surfaced in the Settings â†’ Logs UI as a confirmation-gated button).
- The `logs.recent` publication requires `themeId` in its filter and returns nothing without it. `parseFilter` re-validates client input on the server and clamps the requested limit, so the client can pass an arbitrary filter object without risking injection.

### File Storage Architecture
- Uses `ostrio:files` package for file uploads and storage
- Images stored in `public/.uploads` directory (accessible via `/uploads/` route)
- Storage path uses `process.env.PWD` which works for both dev and mup production deployments
- Directory is automatically created on server startup if it doesn't exist
- Uploaded images generate absolute URLs using `HOST_URL` setting for email compatibility
- Rich text editor integrates file upload directly into TipTap image insertion workflow
